## Pattern analysis

The failures cluster into 3 distinct patterns, each with a different
resolution strategy:

### Pattern A — Skill colors used as FOREGROUND on light surfaces (failures #2, #3, #4, #5)

The fitme-story `--skill-*` tokens were chosen at the Tailwind -500
saturation level for the **PM-flow orbital diagram**, where the skill
color is the FILL of an orbiting node and the surrounding label sits on
the page background, not on the color itself. They were never meant to
function as text/icon foreground on a light surface.

The dashboard's status/priority chips use the color either:
- (a) as background fill with white text on top, OR
- (b) as foreground text on a near-white background.

Pattern (a) is fine — the audit doesn't currently cover it but those
pairs always pass. Pattern (b) is what's failing.

**Resolution (Pattern A → fix):** add `--cr-*-on-light` text-grade tokens
that pick the same Tailwind family but at -700 saturation. The chip
BACKGROUND keeps the -500 color (preserving visual consistency with
the orbital); the chip TEXT uses the -700 variant when displayed on
light backgrounds without a fill.

```css
[data-cr-root] {
  /* Text-grade variants of skill colors for light-mode foreground use.
     Background variants stay --skill-* (-500 family). */
  --cr-status-implementing-text-light: #0369A1; /* sky-700 (was -500 #0EA5E9) */
  --cr-status-done-text-light:         #047857; /* emerald-700 (was -500 #10B981) */
  --cr-priority-high-text-light:       #B45309; /* amber-700 (was -500 #F59E0B) */
  --cr-priority-medium-text-light:     #92400E; /* amber-800 (was amber-400 #FBBF24) */
  /* Dark mode keeps the -500 family — those already pass on dark surfaces. */
}
```

Verified ratios after fix (will re-run audit after T17 lands):

- sky-700 #0369A1 on neutral-50 #FAFAF9 → ~7.5:1 ✓ (AAA)
- emerald-700 #047857 on neutral-50 → ~5.7:1 ✓ (AA)
- amber-700 #B45309 on neutral-50 → ~5.0:1 ✓ (AA)
- amber-800 #92400E on neutral-50 → ~7.7:1 ✓ (AAA)

### Pattern B — Coral as body text (failure #6)

`--color-brand-coral` (#F97066) is fitme-story's accent for **buttons,
headline highlights, and decorative graphics** — never body text. Its
2.67:1 ratio is fine for those uses (button background + white text
gets ~5:1 the other way around).

**Resolution (Pattern B → don't fix; constrain usage):** add a comment to
`token-map.md` noting that `--color-brand-coral` must NOT be used for
body or paragraph text in the control-room. Use `--color-brand-indigo`
for inline links/text accents and reserve coral for buttons/headlines.
Add a lint rule (T17 follow-up) to flag `text-brand-coral` in
control-room JSX.

### Pattern C — Pre-existing fitme-story muted-text gap (failure #7)

`--color-neutral-500` (#78716C) on `--color-neutral-100` (#F5F5F4) is
the standard "muted body text" pair already used throughout the
fitme-story showcase. It misses AA by 0.10 (4.40 vs 4.50). This is a
**pre-existing showcase issue**, not introduced by the dashboard
migration.

**Resolution (Pattern C → don't fix in this PR):** per PRD §6, the
control-room MUST NOT modify shared `--color-*` tokens. For the
control-room specifically, use `--color-neutral-700` (#44403C, ~10:1)
when secondary text would otherwise be hard to read. File a separate
issue to bump fitme-story's `--color-neutral-500` to `#6B635F` (would
hit 4.55:1, AA passing) — that's a showcase-wide change requiring a
separate review.
