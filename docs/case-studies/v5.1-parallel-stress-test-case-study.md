## 4. Analysis

### 4.1 Bottleneck Analysis

| # | Bottleneck | Severity | Rate | Root Cause | Fix |
|---|---|---|---|---|---|
| B1 | `.claude/` write permissions | **Critical** | 16/16 (100%) | Subagent tool permissions don't include `.claude/` paths | Add to allowed write paths |
| B2 | Controller state batch overhead | Medium | ~2 min/phase | Manual state file updates after each batch | Automate if B1 fixed |
| B3 | Agent execution time variance | Low | 7x spread (43s-561s) | More tool uses â‰  better quality | Cap tool uses or set timeout |
| B4 | New directory creation | Low | 1 occurrence | `mkdir` denied for subagents | Controller pre-creates |

### 4.2 What Worked (Zero Issues)

| Aspect | Result | Significance |
|---|---|---|
| **Git merge conflicts** | 0 across 7 phases | Parallel agents use different files/sections |
| **Build failures** | 0 out of 4 attempts | Swift code gen is reliable under parallel load |
| **Same-file edits** | 15 edits, 0 conflicts | MARK sections naturally isolate |
| **pbxproj concurrent edits** | 9 edits, 0 conflicts | Different ID prefixes prevent collision |
| **Quality degradation** | 0 rework needed | v5.1 SoC keeps agent context clean |
| **Type mismatches** | 0 across 15 Swift files | Agents follow existing code patterns |

### 4.3 Critical Finding: Same-File Parallel Writes

Phase 7 proved that 3 agents can edit the same Swift files simultaneously:

- `AnalyticsProvider.swift`: 3 agents added 23 events (10 + 6 + 7) in separate MARK sections
- `AnalyticsService.swift`: 3 agents added 11 convenience methods in separate MARK sections
- `project.pbxproj`: 9 concurrent edits across 5 phases using different ID prefixes (NT, IM, RM, PF)

**Why it worked:** Git's sequential commit model + agents appending at different positions = natural conflict avoidance. This is more resilient than expected, but it's **luck-dependent** â€” agents that modify the same lines would conflict.

## 7. Framework Improvement Signals

### For v5.2

1. **Permission infrastructure** â€” `.claude/` write access for subagents is the single biggest unlock. If fixed, the entire experiment would be fully autonomous.
2. **Pre-flight probes** â€” test capability before committing resources. Maps to branch prediction (Item 6).
3. **Mirror pattern** â€” extract code regions for isolated parallel writes. Maps to systolic chains (Item 7).
4. **Tool use budget** â€” cap subagent tool uses to normalize execution time. Agents that use 68 tools don't produce better output than those using 7.

### Validated v5.1 Principles

| Principle | Validated? | Evidence |
|---|---|---|
| Skill-on-demand (Item 1) | Yes | Agents only loaded relevant context |
| Cache compression (Item 2) | Yes | Agents worked with focused prompts, not full repo |
| Batch dispatch (Item 3) | **Yes** | 4 agents dispatched per phase, controller batched results |
| Result forwarding (Item 4) | Yes | Agent results forwarded to controller for state updates |
| Model tiering (Item 5) | Partially | All agents used Sonnet; haiku would suffice for config tasks |
| Branch prediction (Item 6) | **Research needed** | Pre-flight probe concept identified but not implemented |

## 8. Methodology Notes

- Wall time measured from first dispatch to last commit per phase
- Subagent execution time measured from dispatch to completion notification
- Build times not included in phase duration (run by controller, ~30s each)
- Quality assessed by: does it compile? does it match the spec? does it need rework?
- "Same-file parallel edits" counted as any phase where multiple agents modify the same file
- Controller overhead (state updates, checkpoint saves) counted in phase duration
- All code generated by Sonnet 4.6 subagents; coordination by Opus 4.6 controller
