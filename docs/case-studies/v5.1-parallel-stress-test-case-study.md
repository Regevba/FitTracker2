# Case Study: v5.1 Framework Parallel Stress Test

> **Core question:** How does the PM framework v5.1 (SoC-on-Software) behave under heavy parallel load — 4 features advancing simultaneously through different lifecycle phases?

## 1. Experiment Design

### Independent Variable
Number of parallel PM workflows: **4 concurrent features** (vs typical 1-2)

### Features Under Test

| # | Feature | Starting Phase | Target Phase | Complexity |
|---|---|---|---|---|
| F1 | Push Notifications | PRD | Tasks (or further) | Medium — new iOS capability, permission handling |
| F2 | App Store Assets | PRD | Tasks (or further) | Low — visual assets, no code architecture |
| F3 | Import Training Plan | Research | PRD | High — parser, mapping UI, multi-source import |
| F4 | Smart Reminders | Not started | Research → PRD | High — AI-powered, 5 reminder types, guest overlays |

### Dependent Variables (what we're measuring)

| Metric | How Measured |
|---|---|
| **Throughput** | Features × phases completed per hour |
| **Context efficiency** | Tokens consumed per phase transition |
| **Error rate** | Failed tool calls, build failures, state conflicts |
| **Bottleneck identification** | Which phases slow down under parallel load |
| **Cache hit rate** | L1/L2/L3 cache reuse across features |
| **Cross-feature interference** | State file conflicts, merge conflicts, shared resource contention |
| **Quality degradation** | Spec completeness, review findings, rework needed |

### Controls
- Same agent session (shared context window)
- Same framework version (v5.1)
- Same model (Opus 4.6)
- Sequential phase execution within each feature, parallel across features

### Hypothesis
v5.1's SoC optimizations (skill-on-demand, cache compression, batch dispatch, model tiering) should enable 4 parallel workflows without significant quality degradation. Expected bottlenecks: context window pressure at phase transitions, state file write contention.
